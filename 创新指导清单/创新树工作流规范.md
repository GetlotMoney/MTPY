# DVSR 创新树工作流规范

## 1. 目标

当前阶段的研究目标是先通过“换想法、换模块”把 CUB GZSL 主指标推到 `H >= 74`，达到 74 以后再进入系统调参。

当前严格连续 baseline：

| 数据集 | seed | U | S | H | ZS |
|---|---:|---:|---:|---:|---:|
| CUB GZSL | 5 | 73.30 | 72.53 | 72.91 | 81.72 |

项目正式比较口径：

- 同一设置可以测试多个 seed。
- 主结果不取平均值。
- 主结果按候选 seed 中主指标 `H` 的最大值记录。
- 其它 seed 作为复查记录保留。

## 3. 两层权重

创新树有两层权重：类别权重和模块权重。

类别权重回答：“现阶段这个方向值不值得押注？”

模块权重回答：“这个具体模块本身值不值得试？”

下一步实验选择分：

```text
选择分 = 模块权重 * 类别权重 / 5
```

达到 `H >= 74` 之前，类别权重的默认倾向是：

| 类别 | 当前策略 |
|---|---|
| 局部视觉-属性对齐 | 最高优先级，最贴近 CUB 细粒度瓶颈。 |
| 语义结构/拓扑约束 | 高优先级，保护 seen/unseen 语义流形。 |
| 分支交互/蒸馏 | 高优先级，已有消融证据支持。 |
| 视觉局部编码 | 中高优先级，但避免大改已验证结构。 |
| 训练目标/自监督辅助 | 中高优先级，适合升级 AG-JEPA。 |
| 数据/描述增强 | 中优先级，注意文本漂移。 |
| 推理与校准 | 暂作诊断，不作为主创新。 |
| 评估协议 | 必须保持规范，但不提升 H。 |
| 调参策略 | 达到 74 前低优先级。 |

## 4. 模块权重因子

每个模块用 0 到 5 分评价以下因子：

| 因子 | 含义 |
|---|---|
| `own_gain` | 本项目实验已经证明的收益强度。没有本地结果时为 0。 |
| `paper_gain` | 论文或外部证据支持强度。 |
| `novelty` | 相对当前框架的新颖性。 |
| `compatibility` | 与当前 baseline 兼容程度，越容易插入越高。 |
| `confidence` | 证据可靠性和判断确定性。 |
| `cost` | 实现、训练、调试成本。成本越高越扣分。 |

模块权重公式：

```text
模块权重 =
0.30 * own_gain
+ 0.25 * paper_gain
+ 0.15 * novelty
+ 0.15 * compatibility
+ 0.15 * confidence
- 0.15 * cost
```

## 5. 参杂物/证据来源

每个模块可以挂多种“参杂物”，它们不是装饰，而是用来解释权重为什么这么给。

允许类型：

| 类型 | 用途 |
|---|---|
| `paper` | 论文机制、表格、消融、局限性。 |
| `local_experiment` | 本项目 ABL/MOD/TUN 实验结果。 |
| `user_idea` | 用户提出的研究判断或偏好。 |
| `philosophy` | 可迁移的原则，例如信息瓶颈、流形保持、反事实区分。 |
| `code_observation` | 从当前代码、配置、日志里看到的结构事实。 |

新增想法时，至少要写清楚：

- 它是什么。
- 它为什么可能提升 H。
- 它和当前 baseline 哪个模块连接。
- 它的主要风险是什么。
- 它来自哪些参杂物。

## 6. 状态流转

| 状态 | 含义 |
|---|---|
| `candidate` | 候选想法，还没在本项目实验。 |
| `selected` | 已准备转成下一轮实验。 |
| `testing` | 正在实现或训练。 |
| `validated` | 本地实验支持，应保留或继续扩展。 |
| `weakened` | 有一点价值但弱于预期。 |
| `rejected` | 本地实验不支持，短期不再尝试。 |
| `merged` | 被合并进另一个节点。 |

## 7. 从创新树到实验

每次启动创新实验时：

1. 先看 `创意树.md` 的“下一批尝试建议”。
2. 读取对应分类队列，例如 `创新指导清单/queues/01_module_replacement.md`。
3. 把最高优先级的少量项目同步到 `backlog.md` 当前执行窗口。
4. 从 `backlog.md` 选择当前要执行的实验。
5. 把该节点转成 `MOD-xxx` 实验。
6. 实验代码必须放到 `experiment/innovation` 分支上做。
7. 每轮只改一个主要变量。
8. 新模块必须默认可关闭，配置里要有开关。
9. Claude 审查通过后再跑实验。
10. 训练日志复制到对应实验目录。
11. 结果写入实验 README。
12. 用结果反向更新创新树权重、分类队列和 `backlog.md`。

分类队列和 backlog 的职责边界：

```text
创意树：长期想法库
  ↓
分类待做清单：每类实验自己的候选队列
  ↓
backlog.md：当前执行窗口，只放近期 3-10 个实验
  ↓
experiments/：已经执行或正在执行的实验记录
```

分类队列固定位置：

| 分类 | 队列文件 | 实验目录 |
|---|---|---|
| 创新模块 | `创新指导清单/queues/01_module_replacement.md` | `experiments/01_module_replacement/` |
| 消融实验 | `创新指导清单/queues/02_ablation.md` | `experiments/02_ablation/` |
| 调参实验 | `创新指导清单/queues/03_hyperparam_tuning.md` | `experiments/03_hyperparam_tuning/` |
| 跨数据集 | `创新指导清单/queues/04_cross_dataset.md` | `experiments/04_cross_dataset/` |
| 最终复核 | `创新指导清单/queues/05_final_runs.md` | `experiments/05_final_runs/` |

## 8. 实现前来源复核

把创意变成代码之前，必须先复核创意来源。尤其当模块来自论文时，不能只看创意树摘要就开始写代码。

论文 PDF 固定放在 `创新指导清单/papers/`。文件名优先使用原论文标题，例如：

```text
创新指导清单/papers/TPR Topology-Preserving Reservoirs for Generalized Zero-Shot Learning.pdf
```

如果创意节点引用论文但本地没有 PDF，Codex 必须先尝试查找论文原文；如果访问不到、下载不了、PDF 损坏或无法读取，必须向用户汇报。用户可以手动补充 PDF。为了保证实验流程不断，Codex 可以继续做“非论文原文参考实现”，但必须在实验 README、review packet 和 Claude 审查材料里明确标记：`未参考论文原文`。

来源复核分三层：

| 层级 | 必做事项 |
|---|---|
| 论文机制 | 重新阅读论文中和该模块相关的 method、loss、architecture、ablation 和 limitation。 |
| 官方实现 | 优先查找论文官方 GitHub、作者项目页或 supplementary code。 |
| 替代实现 | 如果没有官方代码，查找高质量复现、同类方法实现或 PyTorch 标准实现，并记录为什么可参考。 |

复核完成后，必须写出“迁移规格”，再允许改代码：

- 原论文模块解决什么问题。
- 原论文代码的输入、输出、张量形状和 loss 如何定义。
- 哪些部分可以迁移到 DVSR。
- 哪些部分不能迁移，为什么。
- DVSR 当前对应的插入点在哪里。
- 新模块关闭时如何退回 baseline。
- 需要新增哪些配置键。
- 预期最小 diff 是哪些文件。
- 是否已读取本地论文 PDF。
- 是否已参考官方代码。
- 如果没有参考论文原文或官方代码，必须写明原因和替代依据。

如果来源代码和论文描述不一致，以代码和论文共同能解释的最小机制为准；如果仍然不确定，允许继续做保守复现，但必须降低 `confidence`、提高 `cost`，并把风险写入创意树和实验 README。

Codex 自审和 Claude 第 1 轮审查都必须检查“来源复核是否充分”。如果完全没有来源复核记录，审查结果应为 `REJECTED`，不能跑训练。如果有复核记录但论文或官方代码访问失败，可以继续实验，但必须标记为 `未参考论文原文` 或 `未参考官方代码`。

## 9. 实验结束后如何更新权重

实验结束后按结果更新节点：

| 结果 | 更新方式 |
|---|---|
| H 明显高于 72.91 | 提高 `own_gain` 和 `confidence`，状态改为 `validated`。 |
| H 小幅接近或略高 | 状态可设为 `weakened` 或 `validated`，但只小幅提高权重。 |
| H 下降但现象有启发 | 状态设为 `weakened`，记录失败原因，保留参杂物。 |
| H 明显下降且无新信息 | 状态设为 `rejected`，降低 `compatibility` 或 `confidence`。 |
| 实现代价远超预期 | 提高 `cost`，不一定降低论文价值。 |

更新时必须记录：

- 实验编号。
- 配置差异。
- seed 和 H。
- 日志路径。
- 和 baseline 的差值。
- 下一步是否继续扩展。

## 11. 维护原则

- 创新树是长期想法库，backlog 是短期执行队列。
- 新想法先进入创新树，不急着写代码。
- 只有机制清楚、风险可控、能写成一个独立实验时，才进入 backlog。
- 本地实验结果比论文声称更重要。
- 当前目标是 `74+`，所以类别和模块权重都要围绕这个目标更新。
- 任何靠换口径得到的“提升”都不能提高模块权重。
