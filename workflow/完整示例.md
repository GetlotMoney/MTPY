## 示例 1：MOD（创新模块）

### 用户指令
```
跑 3 个创新实验
```

### 启动

```
1. 读创意树（创新树）
   └── 外部: 创新指导清单/paper-idea-tree/idea_tree.json
   └── 节点: "类别角度拓扑保持正则"（选择分 2.429）

2. 确定实验
   └── ID: MOD-007
   └── slug: class_angle_topology
   └── 分支: experiment/single-module-innovation → exp/MOD-007_class_angle_topology

3. Git checkpoint
   └── git checkout -b exp/MOD-007_class_angle_topology
   └── git commit -m "checkpoint: before MOD-007 class_angle_topology"
```

### 实验文件

```
experiments/01_single_module_innovation/MOD-007_class_angle_topology/
├── config.yaml              ← 复制 config/VGSR_cub_gzsl.yaml + 新模块开关
├── README.md                ← 模板
├── review-packet.md         ← 审查包
├── framework.md             ← 实验框架图
└── logs/                    ← 训练日志
```

### review-packet.md 内容

```markdown
# MOD-007 审查包

## 实验假设
来自创意树节点: 类别角度拓扑保持正则
- 假设: 约束 seen 类文本两两角度关系，不要偏离 CLIP 原始拓扑

## 来源复核
- 论文: TPR Table 1: best H in 11/12 datasets

## 模块关系
- 与文本拓扑保持: complementary
- 与条件文本扰动: overlap

## 最小 diff
- 新增: `topology_class_angle_loss` 在 `compute_loss` 中
- 配置键: `use_topology_class_angle` (默认 False)

## 退回 baseline
- `use_topology_class_angle=False` → 不计算角度拓扑 loss
```

### framework.md 内容

```markdown
# MOD-007 实验框架图

## 基于哪个 baseline
- baseline: TUNE-024 H=74.09

## 当前实验改动
- 新增模块: 类别角度拓扑保持正则
- 插入点: compute_loss 中，在文本拓扑保持后
- 配置键: `use_topology_class_angle` (默认 False)

## 与 baseline 关系
- 类型: complementary
- 关闭后退回: `use_topology_class_angle=False`

## 模块结构
[文本拓扑保持] → [类别角度拓扑保持] → [总 loss]
```

### 运行

```bash
python train_VGSR_CUB.py --config experiments/01_single_module_innovation/MOD-007_class_angle_topology/config.yaml
```

### 结果

```markdown
## MOD-007 结果

| seed | U | S | H | ZS | 最佳轮次 |
|---:|---:|---:|---:|---:|---:|
| 5 | 74.20 | 74.50 | 74.35 | 82.10 | 48 |

对比:
| 对象 | H | 差值 |
|---|---|---:|
| baseline TUNE-024 | 74.09 | +0.26 |

分级: near_tie
```

### 记录

```
1. 更新 README.md → 写入结果
2. 更新 EXPERIMENT_REGISTRY.md → MOD-007 状态 near_tie
3. 更新创意树 → 节点状态 candidate → tested_neutral
4. 更新 backlog.md → 标记完成
5. Git commit -m "experiment: MOD-007 results, H=74.35"
```

## 示例 4：COMBO（组合模块）

### 用户指令
```
组合 MOD-004 和 MOD-007
```

### 启动

```
1. 读单模块结果
   └── MOD-004: H=72.90 (near_tie)
   └── MOD-007: H=74.35 (near_tie)
   └── 互补假设: MOD-004 提升 U, MOD-007 提升 S

2. 确定实验
   └── ID: COMBO-001
   └── slug: MOD004_MOD007
   └── 分支: experiment/module-combination

3. Git checkpoint
   └── git checkout -b exp/COMBO-001_MOD004_MOD007
```

### 实验文件

```
experiments/02_module_combination/COMBO-001_MOD004_MOD007/
├── config.yaml              ← baseline + MOD-004 + MOD-007
├── README.md
├── review-packet.md
├── framework.md             ← 组合模块图
└── logs/
```

### framework.md 内容

```markdown
# COMBO-001 实验框架图

## 基于哪个 baseline
- baseline: TUNE-024 H=74.09

## 组合模块
- MOD-004: 属性引导局部补丁 OT
  - 插入点: patch 选择后
  - 配置键: `use_attr_patch_ot=True`
- MOD-007: 类别角度拓扑保持
  - 插入点: compute_loss
  - 配置键: `use_topology_class_angle=True`

## 与 baseline 关系
- 打开两个已有模块
- 不改代码，只改 config

## 冲突分析
- MOD-004 改 patch 选择后
- MOD-007 改 loss 计算
- 无位置冲突
```

### 运行

```bash
python train_VGSR_CUB.py --config experiments/02_module_combination/COMBO-001_MOD004_MOD007/config.yaml
```

### 结果

```markdown
## COMBO-001 结果

| 对象 | U | S | H | ZS |
|---|---:|---:|---:|---:|
| baseline | 72.33 | 75.95 | 74.09 | 81.75 |
| MOD-004 单独 | 72.86 | 72.95 | 72.90 | 81.61 |
| MOD-007 单独 | 74.20 | 74.50 | 74.35 | 82.10 |
| COMBO-001 | 74.50 | 76.20 | 75.34 | 82.50 |

combo_gain = 75.34 - max(74.09, 72.90, 74.35) = 75.34 - 74.35 = 0.99

结论: 协同有效（combo_gain > 0.30）
分级: 协同
```

## 每个实验都有的文件

```
experiments/<group>/<EXP-ID>/
├── README.md              ← 实验结果
├── config.yaml            ← 实验配置
├── claude-review.md       ← 审查记录
├── review-packet.md       ← 审查包
├── framework.md           ← 基于哪个 baseline、改了什么
└── logs/                  ← 训练日志
```

**`framework.md` 确保每个实验的 baseline 起点明确，避免对照混乱。**
